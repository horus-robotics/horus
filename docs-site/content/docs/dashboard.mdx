# Dashboard Guide

The HORUS Dashboard is your mission control for robotics applications - monitor performance, debug issues, and tune parameters in real-time from any device on your network.

## Quick Start

### Launch the Dashboard

```bash
# Web dashboard (opens in browser)
horus dashboard

# Custom port
horus dashboard 8080

# Terminal UI mode
horus dashboard --tui
```

**Access from anywhere:**
```
‚Ä¢ Local device:    http://localhost:3000
‚Ä¢ Other devices:   http://192.168.1.100:3000
```

The dashboard automatically binds to all network interfaces, so you can monitor your robot from your phone, tablet, or another computer.

## Common Workflows

### Debugging: Node Not Receiving Messages

**Problem:** Your subscriber node isn't getting data

**Dashboard investigation:**
1. **Nodes Tab** ‚Üí Verify both publisher and subscriber are listed
2. **Topics Tab** ‚Üí Check topic exists and shows >0 subscribers
3. **Graph View** ‚Üí Verify visual connection exists
4. **Logs ‚Üí Filter by node** ‚Üí Check for connection errors

**Common findings:**
- Topic name mismatch (typo in pub/sub)
- Publisher not running
- Shared memory permissions issue
- Namespace confusion

### Performance Optimization

**Problem:** System running slow

**Dashboard investigation:**
1. **Nodes Tab** ‚Üí Sort by CPU (click column header)
2. Identify high CPU nodes
3. Check their tick rates - are they hitting target?
4. **Logs** ‚Üí Filter by that node ‚Üí Look for "slow tick" warnings

**Red flags:**
- Tick rate <50 Hz on 60 Hz node
- CPU >80% sustained
- Memory continuously growing
- Errors increasing rapidly

### Live Parameter Tuning

**Scenario:** Tuning robot motion parameters

**Dashboard workflow:**
1. **Parameters Tab**
2. Find `motion.max_speed`, `motion.acceleration`
3. Edit values while watching robot
4. Check **Nodes Tab** ‚Üí `motion_controller` ‚Üí CPU/tick rate
5. Adjust until behavior is optimal
6. Click "Export Parameters" to save
7. Copy YAML to your config file

## Tips & Best Practices

### Performance Monitoring

**What to watch:**
- Tick rate stability (should be consistent)
- CPU usage trends (spikes are OK, sustained high is bad)
- Memory growth (flat is good, growing is concerning)
- Error count (should be 0 or very slow growth)

**When to investigate:**
- Tick rate drops below 90% of target
- CPU >80% for more than a few seconds
- Memory growing >10 MB/minute
- Error count increasing >1/second

### Parameter Management

**Best practices:**
- Use hierarchical structure: `category.subcategory.param`
- Lowercase with underscores: `max_speed` not `MaxSpeed`
- Descriptive names: `motion.max_linear_velocity` not `motion.vel`
- Export parameters regularly for backup
- Version control your `params.yaml`

### Network Access

**Multi-device monitoring:**

- Remote debugging over VPN

- Only use on trusted networks
- For production: Use localhost only or add auth layer
- Consider firewall rules for external access

## Troubleshooting

### Dashboard shows "No nodes running"

**Cause:** No HORUS application is currently running

**Solution:**
```bash
# Terminal 1 - Start your app
horus run main.rs

# Terminal 2 - Start dashboard
horus dashboard
```

### Logs are empty

**Cause:** Shared memory log buffer doesn't exist

**Solution:**
1. Run a HORUS app first (creates `/dev/shm/horus_logs`)
2. Then start dashboard
3. Refresh browser if needed

### Can't access from phone/tablet

**Cause:** Firewall blocking or wrong IP

**Solution:**
```bash
# 1. Check your computer's IP
ip addr show | grep inet

# 2. Allow port in firewall
sudo ufw allow 3000/tcp

# 3. Access from device
# Use: http://192.168.1.100:3000 (your actual IP)
```

### Port already in use

**Error:** `Port 3000 is already in use`

**Solution:**
```bash
# Use different port
horus dashboard 3001

# Or kill existing process
lsof -ti:3000 | xargs kill -9
```

### Parameters not saving

**Cause:** Config directory doesn't exist or lacks permissions

**Solution:**
```bash
# Create config directory
mkdir -p ~/.horus/config

# Fix permissions
chmod 755 ~/.horus/config
```

## API Reference

For developers building custom integrations, the dashboard provides a REST API:

**Base:** `http://localhost:3000/api`

**Key endpoints:**
- `GET /status` - System health
- `GET /nodes` - All nodes with metrics
- `GET /topics` - Active topics
- `GET /graph` - Node-topic relationships
- `GET /logs/all` - System logs
- `GET /logs/node/:name` - Node-specific logs
- `GET /params` - All parameters
- `POST /params/:key` - Set parameter
- `GET /ws` - WebSocket for real-time updates
